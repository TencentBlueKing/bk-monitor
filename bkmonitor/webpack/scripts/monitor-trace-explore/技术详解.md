## 1. 背景与目标

### 1.1 业务背景

蓝鲸监控平台包含多个子应用，其中 **APM（应用性能管理）** 和 **Trace（链路追踪）** 是两个独立演进的模块：

- **APM 应用**：基于 **Vue 2** 构建，已稳定运行多年，承载服务拓扑、告警、指标等核心功能。
- **Trace 主站**：较新的子应用，基于 **Vue 3** 从零开发，包含 Trace 检索（Trace Explore）、Span 详情、火焰图、时序图等完整的链路分析能力。

产品侧希望在 APM 的服务详情页中直接嵌入 Trace 检索 UI，让用户无需跳转即可从 APM 上下文（服务名、时间范围等）无缝进入 Trace 检索，形成"指标 → 链路"的闭环排查体验。

### 1.2 核心矛盾

这一需求面临的不是单一技术难点，而是多维度的**环境不兼容**：

| 维度 | Trace 主站 | APM 宿主 | 冲突点 |
| --- | --- | --- | --- |
| 框架版本 | Vue 3 | Vue 2 | 运行时完全不兼容，不共享组件实例与响应式系统 |
| 组件库 | bkui-vue（Vue 3） | bkui-vue 2（Vue 2） | 同名不同实现，API/DOM 结构/事件机制均不同 |
| CSS 类名 | `.bk-button`、`.bk-dialog`… | `.bk-button`、`.bk-dialog`… | **完全相同的选择器，完全不同的样式**，同页共存必定互相污染 |
| 构建产物 | webpack 整包 | webpack 整包 | 无法简单 `import` 对方源码，模块解析策略和 loader 链不同 |
| 全局状态 | 自己的 `window.i18n`、路由 | 自己的 `window.i18n`、路由 | 共享 `window` 全局命名空间，初始化可能互相覆盖 |

直接在 APM 中 `import` Trace 源码不可行——Vue 版本冲突会导致编译失败或运行时崩溃；iframe 嵌入则无法共享上下文数据（时间范围、服务筛选等），用户体验割裂。

需要一种方案能同时解决：**跨 Vue 版本运行** + **CSS 样式隔离** + **双向数据通信** + **全局状态不污染**。

### 1.3 解决思路

将 Trace 检索 UI 以 **Vite 库模式（lib mode）** 独立构建为一个**自带运行时的 ES 模块包** `@blueking/monitor-trace-explore`。该包内置独立的 Vue 3 实例、组件库和样式，对外仅暴露 `mount(el, options)` 函数，由 APM 宿主在运行时动态 `import` 并挂载到指定 DOM 节点。

```
┌───────────────────────────────────────────────────────┐
│                   APM 宿主（Vue 2 + webpack）           │
│                                                       │
│  ┌──────────────┐     动态 import     ┌────────────┐  │
│  │ ApmTraceHome │ ──────────────────► │ mount(el)  │  │
│  │  (Vue 2 组件) │ ◄── MountHandle ── │  (Vue 3)   │  │
│  └──────────────┘                     └────────────┘  │
│        ▲ Vue 2 渲染树                   ▲ Vue 3 独立实例 │
│        │                                │              │
│   宿主 webpack 解析                Vite lib 产物        │
│   monitor-trace-explore alias     index.js + chunks   │
└───────────────────────────────────────────────────────┘
```

## 3. Vue 2 ↔ Vue 3 桥接机制

由于 Vue 2 和 Vue 3 运行时完全隔离，需要一套桥接协议在两端传递数据和事件。

### 3.1 桥接流程图

```
  Vue 2 宿主                              Vue 3 子应用
  ──────────                              ──────────
       │                                       │
       │  ① mount(el, { props, onEvent })      │
       │ ─────────────────────────────────────► │
       │                                       │
       │  返回 MountHandle { unmount, update }  │
       │ ◄───────────────────────────────────── │
       │                                       │
       │  ② handle.update({ timeRange: ... })  │
       │ ─────────────────────────────────────► │
       │                     Object.assign(reactive, newProps)
       │                     → Vue 3 自动响应式更新
       │                                       │
       │  ③ bridgeEmit('exploreChartZoomChange', …) 等 │
       │ ◄───────────────────────────────────── │
       │     onEvent(event, ...args)              │
       │                                       │
       │  ④ handle.unmount()                   │
       │ ─────────────────────────────────────► │
       │                     app.unmount()      │
```

### 3.2 TraceExploreApm 适配层（`trace-explore-apm.tsx`）

独立包根组件不再是直接挂载 `TraceExplore`，而是先挂载 **`TraceExploreApm`**：它在 Vue 3 子应用内部承担 **APM 场景适配**，把入口 `mount()` 注入的桥接对象，转换成通用 `TraceExplore` 能消费的 provide/inject 协议。

**组件层级：**

```
trace-explore-apm-entry.ts     ← createApp、provide(BRIDGE_*)、插件注册
  └─ TraceExploreApm           ← inject 桥接、同步 store、provide APM hooks
       └─ TraceExplore          ← 通用检索页；inject 到 hooks 则为 APM 模式，否则为主站模式
```

**职责概要：**

| 能力 | 实现方式 |
| --- | --- |
| 接收宿主属性 | `inject(BRIDGE_PROPS_KEY)`，与入口 `reactive(props)` 对应，`watch(..., { deep: true })` 同步 `timeRange`、`refreshInterval`、`refreshImmediate` 等到 `useTraceExploreStore()` |
| 回传宿主事件 | `inject(BRIDGE_EMIT_KEY)`，封装为 `TraceExploreApmHooks` 并通过 `provide(TRACE_EXPLORE_APM_HOOKS_KEY, …)` 交给 `TraceExplore` 在用户操作时调用 |
| 图表时间轴缩放 | `provide('handleExploreChartZoomChange', fn)`，`fn` 内 `bridgeEmit('exploreChartZoomChange', range)` |
| 子应用根上的 hooks 类型 | `TraceExploreApmHooks`：`onConditionChange`、`onQueryStringChange`、`onFilterModeChange`，内部分别 `bridgeEmit('conditionChange' | 'queryStringChange' | 'filterModeChange', …)` |

**TraceExplore 侧行为：** `inject(TRACE_EXPLORE_APM_HOOKS_KEY, null)` 非空时表示嵌入 APM：检索条件 / 查询语句 / 筛选模式变更会通过上述钩子同步到 Vue 2；同时 **`apmHooks` 为空时**仍渲染主站页头 `TraceExploreHeader`，嵌入时隐藏页头由宿主承接导航。

### 3.3 关键类型定义

```typescript
// Props 方向：Vue 2 → Vue 3（通过 reactive 代理保持响应式）
export interface BridgeProps {
  [key: string]: unknown;
}

// Events 方向：Vue 3 → Vue 2
export type BridgeEmit = (event: string, ...args: unknown[]) => void;

// 入口 provide → TraceExploreApm inject
export const BRIDGE_PROPS_KEY = Symbol('bridgeProps');
export const BRIDGE_EMIT_KEY = Symbol('bridgeEmit');

// TraceExploreApm provide → TraceExplore inject（仅 APM 包挂载链存在）
export const TRACE_EXPLORE_APM_HOOKS_KEY = Symbol('traceExploreApmHooks');
export interface TraceExploreApmHooks {
  onConditionChange?: (condition: IWhereItem[]) => void;
  onQueryStringChange?: (queryString: string) => void;
  onFilterModeChange?: (mode: EMode) => void;
}

// mount() 返回的控制句柄
export interface MountHandle {
  unmount: () => void;
  update: (newProps: Partial<BridgeProps>) => void;
}
```

### 3.4 宿主侧组件实现

宿主中有两层 Vue 2 组件：

| 组件                    | 文件                                  | 职责                                                                                                |
| ----------------------- | ------------------------------------- | --------------------------------------------------------------------------------------------------- |
| `ApmTraceHome`          | `apm-trace-explore/index.tsx`         | 注入 viewOptions、timeRange 等业务上下文                                                            |
| `TraceExploreContainer` | `apm-trace-explore/trace-explore.tsx` | 动态 import 子应用、调用 `mount()`、Watch props 变更调用 `update()`、`beforeDestroy` 时 `unmount()` |

其中 `TraceExploreContainer` 在 `mounted()` 阶段会临时保存并恢复 `window.i18n`，防止 Vue 3 子应用的 i18n 初始化污染宿主全局 i18n 实例。

```
│                                                                │

└────────────────────────────────────────────────────────────────┘
```

**fork-mermaid alias 的关键性**：

fork-mermaid 发布两种 dist 格式：

| 入口                       | sequence diagram 的 d3 来源                                 | 效果                                  |
| -------------------------- | ----------------------------------------------------------- | ------------------------------------- |
| `mermaid.core.mjs`（默认） | `import { select } from "d3"` — 裸路径                      | 宿主 webpack 可能解析到错误/缺失的 d3 |
| `mermaid.esm.mjs`          | `import { q as select } from "./mermaid-xxx.js"` — 相对路径 | d3 自包含，永远正确                   |

通过 alias 指向 `mermaid.esm.mjs`，确保 d3 依赖自包含在产物中，不依赖宿主环境。

### 4.3 external 依赖

以下模块不打入库产物，交由宿主 webpack 解析：

```javascript
external: [
  /^dayjs[/]?\w*/, // 日期库
  '@prometheus-io/lezer-promql', // PromQL 语法解析
  /^monaco-editor[/]?\w*/, // 编辑器
  /^monitor-api\/.*/, // 监控 API 层
  /^monitor-common\/.*/, // 公共工具
  /^monitor-pc\/.*/, // 监控 PC 模块
  /^monitor-ui\/.*/, // 监控 UI 组件
  /^monitor-static\/.*/, // 静态资源
];
```

### 4.4 chunk 文件扩展名：`.js` 而非 `.mjs`

宿主 webpack 在解析 `.mjs` 文件的 `import` 时会启用 `fullySpecified` 模式，要求所有裸路径也带扩展名，而工程别名解析的是无后缀的 `.ts`/`.js`，会整包报错。使用 `.js` 扩展名可按 `javascript/auto` 处理，兼容宿主解析逻辑。

## 5. CSS 类名前缀隔离

### 5.1 问题

宿主 APM（Vue 2）和 Trace 子应用（Vue 3）都使用 `.bk-*` 作为组件类名前缀，但对应完全不同的组件库样式，会互相污染。

### 5.2 解决方案

通过**构建时**和**运行时**两个阶段、三条路径协同完成隔离：

```
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                          构建时 (Vite build)                            │
  │                                                                         │
  │  ┌──────────────────────────────────┐  ┌──────────────────────────────┐ │
  │  │ ① apmBkClassPrefixPostcssPlugin │  │ ② apmBkClassPrefixSource-   │ │
  │  │                                  │  │    Plugin                    │ │
  │  │  作用对象：bkui-vue 的 CSS        │  │                              │ │
  │  │  变换规则：.bk-xxx → .apm-bk-xxx │  │  作用对象：业务源码            │ │
  │  │  判定条件：filePath 含 bkui-vue   │  │           (非 node_modules)  │ │
  │  │                                  │  │  变换规则：bk-xxx → apm-bk-  │ │
  │  │  确保组件库 CSS 选择器带前缀      │  │  确保业务代码中的类名引用      │ │
  │  │                                  │  │  与 CSS 选择器匹配            │ │
  │  └──────────────────────────────────┘  └──────────────────────────────┘ │
  └─────────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────────┐
  │                         运行时 (trace-explore-apm-entry.ts)             │
  │                                                                         │
  │  ┌──────────────────────────────────────────────────────────────────┐   │
  │  │ ③ 入口文件直接修改 bkui-vue 默认配置                             │   │
  │  │                                                                  │   │
  │  │  defaultRootConfig.prefix = 'apm-bk'                            │   │
  │  │    └→ 覆盖 usePrefix() 的 inject 回退值                          │   │
  │  │    └→ 确保所有组件（含 $bkPopover 等独立实例）生成 .apm-bk-* 类名 │   │
  │  │                                                                  │   │
  │  │  provideGlobalConfig({ prefix: 'apm-bk' }, app)                 │   │
  │  │    └→ 通过 provide/inject 为主 app 组件树注入前缀                 │   │
  │  │    └→ 设置 CSS 变量 --bk-prefix = 'apm-bk'                      │   │
  │  └──────────────────────────────────────────────────────────────────┘   │
  └─────────────────────────────────────────────────────────────────────────┘
```

### 5.3 三条路径各自的必要性

| 路径 | 覆盖范围 | 缺失后果 |
|------|---------|---------|
| ① PostCSS 插件 | bkui-vue 编译产出的 CSS 选择器 | 组件库样式仍为 `.bk-*`，与宿主冲突 |
| ② Source 插件 | 业务 TS/TSX/SCSS 中硬编码的 `bk-` 类名 token | 业务代码引用的类名与 CSS 不匹配 |
| ③ 运行时 `defaultRootConfig.prefix` | bkui-vue JS 运行时生成的 DOM class 属性 | 组件渲染出 `.bk-*` 类名，但 CSS 已变为 `.apm-bk-*` |

### 5.4 Source 插件的正则设计

```typescript
const apmBkClassTokenRe = /(?<![</\w-])bk-(?=[a-z0-9])/g;
```

| 正则部分 | 含义 | 避免误伤 |
|---------|------|---------|
| `(?<![</\w-])` | 负向后行断言：前一个字符不能是 `<`、`/`、字母数字下划线、`-` | 不替换 HTML 标签名（如 `<bk-xxx>`）、已带前缀的 `apm-bk-`、以及单词内部的 `bk-` |
| `bk-` | 匹配 bk- 前缀本身 | — |
| `(?=[a-z0-9])` | 正向先行断言：后面必须跟小写字母或数字 | 避免匹配 `bk-` 作为其他用途（如注释中的文字） |

## 6. 条件编译（ifdef）

### 6.1 机制概述

项目同时在两种构建中使用同一份源码：

| 构建       | 工具          | IS_APM_MONITOR | 用途      |
| ---------- | ------------- | -------------- | --------- |
| Trace 主站 | webpack       | `false`        | 完整功能  |
| APM 嵌入库 | Vite lib mode | `true`         | 裁剪/适配 |

### 6.2 语法

| 场景 | 条件宏写法 | 编译后保留 |
| ---- | ---------- | ---------- |
| TS / TSX | `// #if IS_APM_MONITOR` … `// #endif` | 仅 **APM 库**构建（`IS_APM_MONITOR === true`） |
| JSX（TSX 内嵌） | `{/* // #if IS_APM_MONITOR */}` … `{/* // #endif */}` | 与上表同一套宏；指令写在 JSX 块注释里，避免破坏 TSX 语法 |
| SCSS | `// #if IS_APM_MONITOR` … `// #endif`（与 TS 相同的 `//` 单行注释形式） | 与 TS 一致；**必须在 Sass 之前**

### 6.3 双构建 ifdef 处理流程

```
             Vite 构建 (build:trace-explore)
             ─────────────────────────────
  源码 (.tsx/.scss)
    │
    ▼ ifdefPlugin (enforce: 'pre')  ← Sass 之前执行，保留 #if 指令
    │   IS_APM_MONITOR = true
    │   JSX 注释块 → 独立行
    │   ifdef-loader/preprocessor.parse()
    ▼
  经条件编译的源码 → Sass/TS 编译 → 产物


             webpack 构建 (trace 主站)
             ─────────────────────────
  源码 (.tsx/.scss)
    │
    ▼ createTraceWebpackIfdefRules (enforce: 'pre')
    │   TS/TSX: ifdef-loader + jsx-ifdef-normalize-loader
    │   SCSS:   ifdef-loader (单独规则)
    │   IS_APM_MONITOR = false
    ▼
  经条件编译的源码 → webpack 正常流程 → 产物
```

**重要细节**：SCSS 中 `// #if` 是合法的单行注释，会被 Sass 编译器直接删除。必须在 Sass 之前执行 ifdef 处理（Vite 用 `enforce: 'pre'`；webpack 用 `enforce: 'pre'` 规则），否则条件编译在 SCSS 中永远不生效。

### 6.4 实际应用场景

- **v-overflow-tips 样式修复**：APM 宿主的全局 `.tippy-*` 样式会污染 Trace 组件的 tooltip，通过 ifdef 注入额外的箭头定位修正
- **tippy 弹层箭头微调**：宿主环境与主站的 tippy 渲染位置有微小差异，需要条件性的 margin 修正

## 7. 关键问题排查与解决

### 7.1 mermaid getBBox 报错

**现象**：

```
TypeError: Cannot read properties of null (reading 'getBBox')
    at drawActorTypeParticipant (sequenceDiagram-*.js)
```

**根因链**：

```
trace-explore-apm-entry
  └→ TraceExploreApm → TraceExplore → explore-trace-slider → TraceDetail → SequenceGraph
       └→ import 'fork-mermaid'
            └→ mermaid.render(id, text)
                 └→ select("body").append("div")  ← d3 空 selection！
                      └→ draw() 所有 SVG 操作在空 selection 上执行
                           └→ .node() 为 null → getBBox() 报 TypeError
```

**根本原因**：

在 Vite lib mode 构建 + 宿主 webpack 二次消费的场景下，mermaid 内部通过 `select("body")` 或 `select('[id="..."]')` 查找/创建 SVG 容器时返回空 selection。

**解决方案（两步）**：

1. **`defineAsyncComponent` 懒加载 SequenceGraph**：

```typescript
// trace-detail.tsx
const SequenceGraph = defineAsyncComponent(() => import('../../../plugins/charts/sequence-graph/sequence-graph'));
```

初始加载不拉入 fork-mermaid，只在用户点击「时序图」Tab 时才按需加载。

2. **手动创建离屏 DOM 容器传给 `mermaid.render()`**：

```typescript
// sequence-graph.tsx
const renderContainer = document.createElement('div');
renderContainer.style.cssText = 'position:fixed;top:-9999px;left:-9999px;width:2000px;height:2000px;visibility:hidden';
document.body.appendChild(renderContainer);
try {
  ({ svg } = await mermaid.render(graphId, graphDefinition.value, renderContainer));
} finally {
  renderContainer.remove();
}
```

传入 `svgContainingElement` 后，mermaid 内部改用 `select(element)`（直接包装 DOM 节点，不经过 CSS 选择器），规避宿主环境差异。容器需有实际尺寸且挂在 DOM 树上，否则 SVG 的 getBBox / 布局计算无法得到正确结果。

3. **fork-mermaid alias 指向 `mermaid.esm.mjs`**：

解决 d3 裸路径 bare specifier 在宿主 webpack 中解析失败的问题。

### 7.2 d3-transition 缺失

**现象**：

```
TypeError: g2.transition is not a function
```

**根因**：`use-flame.ts` 导入了 `d3-selection` 并调用 `.transition()`，但未显式 import `d3-transition`。D3 模块化架构中，`d3-transition` 作为 side-effect 包扩展 `Selection.prototype.transition` 方法，必须显式导入。

**修复**：

```typescript
// use-flame.ts
import 'd3-transition'; // ← 新增 side-effect import
```

### 7.3 $bkPopover 在 APM 环境失效

**现象**：`$bkPopover` 创建的弹窗无法弹出。

**根因分析**：

```
$bkPopover → createApp() → 独立 Vue 实例（不在主 app provide 树中）
  └→ usePrefix() → inject(prefixKey) → 回退到 defaultRootConfig.prefix
       └→ 如果 defaultRootConfig.prefix 仍为 'bk'，则生成 .bk-* 类名
            └→ 但 CSS 已被改为 .apm-bk-*，类名不匹配 → 样式失效
```

**解决方案**：

嵌入场景中 `$bkPopover` 依赖 bkui-vue 内部实现`usePrefix()`，在宿主环境中行为不可预测；统一使用原生 tippy.js 更可控。

### 7.4 tippy.js 弹窗样式被宿主污染

**现象**：`v-overflow-tips` 和统计弹窗在 APM 环境中样式错乱。

**根因**：APM 宿主的全局 CSS 中也有 `.tippy-box` 和 `[data-theme~='light']` 的样式规则，与 Trace 组件的 tippy 样式冲突。

**解决方案**：

1. **将 `$bkPopover` 替换为原生 tippy.js**：`$bkPopover` 是 bkui-vue 封装，依赖 bkui-vue 内部 DOM 结构和前缀；直接使用 tippy.js 更可控。

2. **双 theme token 提高 CSS 特异性**：

```scss
// 同时带 light 与自定义 token，特异性高于宿主全局样式
.tippy-box[data-theme~='light'][data-theme~='statistics-dimension-popover-cls'] {
  width: 405px;
  border: 1px solid #dcdee5;
  .tippy-content {
    padding: 12px;
  }
}
```

3. **条件编译微调箭头定位**：不同宿主环境的 tippy 渲染位置略有差异，通过 `// #if IS_APM_MONITOR` 注入修正样式。

## 8. bk-user-display-name 自定义元素适配

### 8.1 背景

项目中广泛使用 `<bk-user-display-name>` 来展示用户信息（用户名 → 显示名映射）。它是通过 `@blueking/bk-user-display-name` 注册的 **Web Components 自定义元素**（Custom Element），不属于 Vue 组件体系。

在独立包场景下需要处理两个问题：

1. **Vue 编译器误识别**：Vue 3 模板编译器遇到未知标签时会调用 `resolveComponent()` 尝试从组件注册表中查找，找不到则抛出 warning 甚至报错。
2. **自定义元素需初始化**：`<bk-user-display-name>` 需要调用 `UserDisplayName.configure()` 注入 API 地址、租户 ID 等配置后才能正常工作。

### 8.2 解决方案

需要在**构建时**和**运行时**两处同时配置：

```
                 构建时                                运行时
                 ──────                                ──────
  ┌───────────────────────────────────┐  ┌────────────────────────────────────┐
  │ Vite 配置 (build.ts)             │  │ mount() 入口                       │
  │                                   │  │                                    │
  │ vueTsx({                          │  │ ① userDisplayNameConfigure()       │
  │   isCustomElement:                │  │   └→ 调用 UserDisplayName          │
  │     tag => tag ===                │  │      .configure({ tenantId,       │
  │       'bk-user-display-name'      │  │        apiBaseUrl, ... })          │
  │ })                                │  │   └→ 注册自定义元素到浏览器         │
  │                                   │  │                                    │
  │ 告诉 Vue 编译器：                  │  │ ② app.config.compilerOptions      │
  │ 这是原生自定义元素，               │  │     .isCustomElement =             │
  │ 不要 resolveComponent             │  │     tag => tag ===                 │
  │                                   │  │       'bk-user-display-name'       │
  └───────────────────────────────────┘  │                                    │
                                         │   运行时再次声明，确保运行时         │
                                         │   编译器（如 v-if 动态分支）        │
                                         │   也能正确识别                      │
                                         └────────────────────────────────────┘
```

### 8.3 构建时：`@vitejs/plugin-vue-jsx` 配置

```typescript
// build.ts
vueTsx({
  isCustomElement: tag => tag === 'bk-user-display-name',
})
```

Vue JSX 插件在编译 TSX 时，遇到 `<bk-user-display-name>` 会直接生成 `document.createElement('bk-user-display-name')` 调用，而非 `resolveComponent('bk-user-display-name')`。

### 8.4 运行时：初始化与声明

```typescript
// trace-explore-apm-entry.ts
export function mount(el, options) {
  // ① 注册自定义元素并配置 API
  userDisplayNameConfigure();

  const app = createApp(TraceExploreApm);

  // ② 运行时编译器也标记为自定义元素
  app.config.compilerOptions = {
    ...app.config.compilerOptions,
    isCustomElement: tag => tag === 'bk-user-display-name',
  };

  // ...
}
```

`userDisplayNameConfigure()` 内部调用 `UserDisplayName.configure()` 完成：
- 将 `<bk-user-display-name>` 注册到浏览器的 `customElements` 注册表
- 注入 API 基础路径（根据是否多租户模式区分）、租户 ID、缓存时长等配置

## 9. 一些重要细节

1. **i18n 隔离**：子应用挂载时会设置 `window.i18n`，宿主在 `mount()` 前后需保存和恢复自身的 `window.i18n`。
   关键代码：

```ts
// ...
const savedI18n = window.i18n;
// 本地测试使用
const { mount } = await import('monitor-trace-explore');
// 线上使用
// const { mount } = await import('@blueking/monitor-trace-explore');
window.i18n = savedI18n;
// ...
```

2. **CSS 作用域**：所有 bkui-vue 的 CSS 类名在构建时被加上 `apm-` 前缀（`.bk-*` → `.apm-bk-*`），业务自定义类名不受影响。
   关键代码：

```ts
export default defineConfig({
  // ...
  plugins: [
    apmBkClassPrefixSourcePlugin(), // 业务源码中的 bk- 类名 token
  ],
  css: {
    postcss: {
      plugins: [apmBkClassPrefixPostcssPlugin()], // bkui-vue CSS 选择器
    },
  },
  // ...
});
// 入口文件中：
defaultRootConfig.prefix = 'apm-bk'; // 运行时兜底
```

3. **SCSS ifdef 须在 Sass 之前**：`// #if` 在 SCSS 中是注释，Sass 编译后会被删除。无论 Vite 还是 webpack，ifdef 处理必须在 Sass 之前执行。
   关键代码：

```ts
function ifdefPlugin(): VitePlugin {
  // ...
  return {
    name: 'vite-plugin-ifdef',
    /**
     * 必须在 Sass/Less 等预处理器之前执行：`// #if` 在 SCSS 里是合法注释，若先被编译，
     * 指令行会被删掉而块内样式仍保留，条件编译会「永远不生效」。
     */
    enforce: 'pre',
    transform(code: string, id: string) {
      // ...
    },
  };
}
```

4. **fork-mermaid 必须用 esm 入口**：`mermaid.core.mjs` 中 d3 使用裸路径 import，在宿主 webpack 环境中会解析失败；`mermaid.esm.mjs` 的 d3 自包含，永远正确。
   关键代码：

```ts
export default defineConfig({
  // ...
  resolve: {
    alias: [
      // ...
      /**
       * fork-mermaid 发布了两种 dist 格式：
       * 入口	sequence diagram 的 select 来源	效果
       * mermaid.core.mjs (默认)
       * import { select } from "d3" — 裸路径，由宿主 webpack 解析，宿主可能解析到错误/缺失的 d3
       * mermaid.esm.mjs
       * import { q as select } from "./mermaid-aafb0501.js" — 相对路径，同 bundle，d3 自包含，永远正确
       * 之前无论是 Vite 打包还是外置到宿主 webpack，
       * 序列图 chunk 里的 import { select } from "d3" 始终作为一个裸路径 bare specifier 被宿主 webpack 独立解析。
       * 宿主 Vue2 APM 项目如果没有安装 d3 或版本不匹配，
       * select 函数就会异常，导致所有 d3 DOM 操作返回空 selection，.node() 为 null，最终 getBBox() 报错。
       */
      {
        find: 'fork-mermaid',
        replacement: resolve(__dirname, '../../src/trace/node_modules/fork-mermaid/dist/mermaid.esm.mjs'),
      },
    ],
  },
  // ...
});
```

## 10. 构建与开发指南

### 10.1 构建独立包

```bash
# 在 bkmonitor/webpack 目录下
yarn build:trace-explore
```

产物输出到 `monitor-trace-explore/`。

### 10.2 宿主集成

1. 在宿主 `webpack.config.js` 中配置 alias：

```javascript
resolve: {
  alias: {
    'monitor-trace-explore': path.resolve(__dirname, './monitor-trace-explore/index.js'),
  }
}
```

2. 安装 `monitor-trace-explore/package.json` 中声明的 `dependencies`。

3. 在宿主 Vue 2 组件中使用：

```javascript
const { mount } = await import('monitor-trace-explore');
const handle = mount(el, {
  props: { viewOptions, timeRange, refreshInterval },
  onEvent: (event, ...args) => {
    /* 处理 Vue 3 抛出的事件 */
  },
});
// 后续更新 props
handle.update({ timeRange: newRange });
// 销毁
handle.unmount();
```

`onEvent` 常见事件名（由 `TraceExploreApm` 经 `bridgeEmit` 抛出，与业务约定一致即可）：`conditionChange`、`queryStringChange`、`filterModeChange`（检索条件 / 查询语句 / UI·语句模式同步回宿主）、`exploreChartZoomChange`（图表时间范围缩放）。`props` 除示例外还可包含 `queryString`、`refreshImmediate` 等，由适配层写入 store 或 `window.APM_QUERY_STRING`。

### 10.3 本地开发联调

本地开发时 Trace 主站仍使用 webpack 构建（`yarn dev`），ifdef 值为 `IS_APM_MONITOR: false`，所有功能正常。需要测试 APM 嵌入效果时：

1. 运行 `yarn build:trace-explore` 生成产物
2. 在 APM 宿主项目中引用产物路径
3. 启动 APM 宿主的 dev server
