# 项目目标

## 背景

bkmonitor 的数据接入、查询和存储链路中，DataSource、ResultTable、字段、存储、空间授权、VM 接入等信息分散在多个模型和运行时配置中。排查问题时通常需要围绕 `bk_data_id`、`table_id`、`data_label`、`space_uid` 等标识跨模型查询。

`bkmonitor-admin` 的目标是提供一个独立管理后台，把这些核心资源及其关联关系以可检索、可跳转、可对照的方式展示出来。

这个项目还需要天然适合 AI 协作：一方面代码主要由 AI 助手长期参与实现，另一方面后续会加入 AI 辅助排障功能，因此资源模型、接口协议、错误结构和文档都要为 Agent 调用做好准备。

## 一期目标

第一期只做信息展示：

- 提供左侧资源类型导航，右侧展示对应资源列表与详情。
- 支持 DataSource 与 ResultTable 两类核心资源。
- 支持多套 bkmonitor 环境配置和切换。
- 支持围绕核心字段的检索、过滤、排序和分页。
- 支持从一个资源跳转到关联资源，例如从 DataSource 跳到 ResultTable，从 ResultTable 跳到 DataSource / ESStorage / VM 记录。
- 对字段数量巨大的 ResultTableField 做懒加载、分页和局部检索，不在 ResultTable 列表中展开字段明细。
- 所有展示接口优先通过 `kernel_api/rpc` 提供，前端不直接理解 Django ORM 细节。
- 所有请求都必须绑定当前环境，避免多套 bkmonitor 平台之间的缓存和操作混淆。
- API 设计需要有机器可读 schema、稳定命名、示例参数和安全级别，便于未来 AI Agent 调用。

## 非目标

第一期不处理：

- AI 自动诊断和自动执行。
- 新增、编辑、删除、刷新路由等写操作。
- 直接连接数据库或绕过 bkmonitor 后端鉴权。
- 把现有 bkmonitor 主前端迁移到新项目。

## 产品形态

页面结构采用传统 admin 布局：

- 顶部：环境信息、租户信息、当前用户、全局搜索入口。
- 左侧：资源类型导航。
- 右侧：当前资源的列表、过滤器、详情抽屉或详情页。

一期建议的导航：

- 数据源 DataSource
- 结果表 ResultTable
- 字段 Field
- 存储 Storage
- VM 接入

其中 Field、Storage、VM 接入可以先不做独立入口，但需要作为 ResultTable 详情的关联面板出现。

## 设计原则

- 以 bkmonitor 的真实模型关系为准，不在前端发明新的资源语义。
- 列表展示轻量字段，详情展示完整信息，关联信息按需加载。
- 所有查询必须带租户上下文，不能让前端产生跨租户模糊查询。
- 资源 ID 可复制、可跳转、可作为全局搜索输入。
- 默认只读；未来写操作必须有权限、确认、审计和回滚提示。
